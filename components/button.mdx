## Overview

Buttons are interactive elements that allow users to perform actions and make choices with a single tap or click. They communicate actions that users can take and are typically placed throughout your UI, in places like dialogs, forms, cards, and toolbars.

## When to Use

<CardGroup cols={2}>
  <Card title="Primary Actions" icon="star">
    Use for the most important action on a page or section
  </Card>

  <Card title="Form Submissions" icon="paper-plane">
    Submit forms, complete flows, or confirm actions
  </Card>

  <Card title="Navigation" icon="arrow-right">
    Move users to different pages or sections
  </Card>

  <Card title="Triggering Actions" icon="play">
    Start processes, open dialogs, or execute commands
  </Card>
</CardGroup>

### When Not to Use

<Warning>
  - **Navigation within text**: Use links instead
  - **Too many actions**: Prioritize and use secondary buttons or links for less important actions
  - **Non-interactive elements**: Don't style non-clickable elements as buttons
  - **Toggle states**: Use toggle switches or checkboxes instead
</Warning>

## Variants

### Primary Button

The primary button is used for the main call-to-action on a page or within a section. Use sparingly—typically only one primary button per screen or section.

**Use for:**
- Main action on a page (e.g., "Save", "Submit", "Continue")
- High-emphasis actions that advance the user's workflow
- Final step in a multi-step process

### Secondary Button

Secondary buttons are used for important but not primary actions. They provide visual weight but don't compete with the primary action.

**Use for:**
- Alternative actions (e.g., "Cancel", "Go Back")
- Actions of medium emphasis
- Multiple actions of equal importance

### Tertiary Button

Tertiary buttons are the lowest emphasis variant. They're used for less important actions or when multiple actions are needed.

**Use for:**
- Optional actions
- Less frequently used actions
- Actions in data-dense interfaces (tables, lists)

### Ghost Button

Ghost buttons have minimal visual weight and are used for the least important actions.

**Use for:**
- Dismissive actions (e.g., "Cancel", "Skip")
- Actions within other components (cards, banners)
- Tertiary navigation

### Icon Button

Icon-only buttons are compact and used when space is limited or the icon clearly communicates the action.

**Use for:**
- Common actions with universally understood icons (e.g., close, edit, delete)
- Toolbars and action bars
- Compact interfaces

<Note>
  Always include an accessible label for icon buttons, even if it's not visually displayed.
</Note>

## States

All button variants support the following states:

### Default
The standard appearance when the button is ready for interaction.

### Hover
Visual feedback when the user's cursor is over the button.

### Focus
Indicates the button has keyboard focus, important for accessibility.

### Active/Pressed
The appearance while the button is being clicked or pressed.

### Disabled
Indicates the action is currently unavailable. Include a tooltip explaining why when possible.

### Loading
Shows that an action is in progress. The button should remain disabled during this state.

## Design Specifications

### Spacing

- **Padding (Medium)**: 12px top/bottom, 24px left/right
- **Gap**: 8px between icon and text
- **Minimum width**: 88px (to ensure touch target size)
- **Border radius**: 8px
- **Icon size**: 20px × 20px

### Colors

#### Primary Button
- **Background**: `primary-600` (#16A34A)
- **Background (Hover)**: `primary-700` (#15803D)
- **Background (Active)**: `primary-800` (#166534)
- **Text**: `white` (#FFFFFF)
- **Border**: None

#### Secondary Button
- **Background**: `transparent`
- **Background (Hover)**: `primary-50` (#F0FDF4)
- **Background (Active)**: `primary-100` (#DCFCE7)
- **Text**: `primary-700` (#15803D)
- **Border**: 1px solid `primary-600` (#16A34A)

#### Disabled State (All variants)
- **Background**: `gray-100` (#F3F4F6)
- **Text**: `gray-400` (#9CA3AF)
- **Opacity**: 0.5
- **Cursor**: not-allowed

### Typography

- **Font family**: Inter
- **Font weight**: 600 (Semi-bold)
- **Font size**:
  - Small: 14px
  - Medium: 16px
  - Large: 18px
- **Line height**: 1.5
- **Text transform**: None
- **Letter spacing**: 0

### Effects

- **Focus ring**: 2px solid `primary-500`, 2px offset
- **Shadow (Default)**:
  - Offset: 0px 1px 2px
  - Color: rgba(0, 0, 0, 0.05)
  - Blur: 2px
- **Shadow (Hover)**:
  - Offset: 0px 2px 4px
  - Color: rgba(0, 0, 0, 0.1)
  - Blur: 4px
- **Transition**: all 150ms ease-in-out

## Accessibility

### WCAG Compliance

<AccordionGroup>
  <Accordion title="Keyboard Support" icon="keyboard">
    - **Tab**: Move focus to the button
    - **Shift + Tab**: Move focus away from the button
    - **Enter/Space**: Activate the button
    - **Disabled buttons**: Should not receive keyboard focus
  </Accordion>

  <Accordion title="Screen Reader Support" icon="audio-description">
    - Use semantic `<button>` elements
    - Include descriptive labels (avoid "Click here")
    - For icon buttons, provide `aria-label`
    - Use `aria-disabled="true"` for disabled state (not just visual styling)
    - Add `aria-busy="true"` during loading states
  </Accordion>

  <Accordion title="Color Contrast" icon="eye">
    - **Primary button text**: 4.5:1 minimum contrast ratio ✓
    - **Secondary button text**: 4.5:1 minimum contrast ratio ✓
    - **Focus indicator**: 3:1 minimum contrast ratio ✓
    - Don't rely on color alone to convey state
  </Accordion>

  <Accordion title="Touch Targets" icon="hand-pointer">
    - **Minimum size**: 44px × 44px (WCAG 2.5.5)
    - Ensure adequate spacing between adjacent buttons (8px minimum)
    - Large buttons recommended for mobile interfaces
  </Accordion>
</AccordionGroup>

### ARIA Attributes

```html
<!-- Standard button -->
<button type="button">
  Save Changes
</button>

<!-- Icon button -->
<button type="button" aria-label="Close dialog">
  <CloseIcon />
</button>

<!-- Disabled button -->
<button type="button" aria-disabled="true" disabled>
  Submit
</button>

<!-- Loading button -->
<button type="button" aria-busy="true" disabled>
  <Spinner /> Processing...
</button>

<!-- Button that controls another element -->
<button
  type="button"
  aria-expanded="false"
  aria-controls="menu-1"
>
  Menu
</button>
```

## Usage Examples

### React/TypeScript

```tsx
import { Button } from '@/components/ui/button';

// Primary button
<Button variant="primary" size="medium">
  Save Changes
</Button>

// Secondary with icon
<Button variant="secondary" size="medium" icon={<PlusIcon />}>
  Add Item
</Button>

// Loading state
<Button variant="primary" loading disabled>
  Processing...
</Button>

// Disabled
<Button variant="primary" disabled>
  Submit
</Button>

// Full width
<Button variant="primary" fullWidth>
  Continue
</Button>

// Icon only
<Button
  variant="ghost"
  size="small"
  iconOnly
  aria-label="Delete item"
>
  <TrashIcon />
</Button>
```

### HTML/CSS

```html
<!-- Primary button -->
<button class="btn btn-primary btn-medium">
  Save Changes
</button>

<!-- Secondary button -->
<button class="btn btn-secondary btn-medium">
```

## Best Practices

<AccordionGroup>
  <Accordion title="Button Labels" icon="tag">
    ✅ **Do:**
    - Use clear, action-oriented labels ("Save", "Delete", "Submit")
    - Keep labels concise (1-3 words)
    - Use sentence case
    - Be specific about the action ("Delete Account" vs "Delete")

    ❌ **Don't:**
    - Use vague labels ("OK", "Click here")
    - Write full sentences
    - Use ALL CAPS
    - Make users guess what will happen
  </Accordion>

  <Accordion title="Button Hierarchy" icon="layer-group">
    ✅ **Do:**
    - Use one primary button per screen/section
    - Place primary button on the right in Western interfaces
    - Use secondary/tertiary for less important actions
    - Maintain consistent placement across the app

    ❌ **Don't:**
    - Use multiple primary buttons competing for attention
    - Mix different button styles for the same action type
    - Bury important actions in tertiary buttons
  </Accordion>

  <Accordion title="Loading States" icon="spinner">
    ✅ **Do:**
    - Disable button during loading
    - Show loading indicator (spinner)
    - Keep button label visible when possible
    - Provide feedback when action completes

    ❌ **Don't:**
    - Allow multiple clicks during processing
    - Remove all button content during loading
    - Leave users wondering if action worked
  </Accordion>

  <Accordion title="Disabled State" icon="ban">
    ✅ **Do:**
    - Use sparingly
    - Provide tooltip explaining why it's disabled
    - Enable as soon as conditions are met
    - Consider hiding instead if never relevant

    ❌ **Don't:**
    - Disable without explanation
    - Use as the default state
    - Permanently disable interactive elements
  </Accordion>
</AccordionGroup>

## Common Patterns

### Form Actions

```tsx
<div className="form-actions">
  <Button variant="secondary" onClick={handleCancel}>
    Cancel
  </Button>
  <Button variant="primary" type="submit">
    Save Changes
  </Button>
</div>
```

### Confirmation Dialog

```tsx
<Dialog>
  <DialogContent>
    <DialogTitle>Delete Account</DialogTitle>
    <DialogDescription>
      This action cannot be undone. Are you sure?
    </DialogDescription>
    <DialogActions>
      <Button variant="ghost" onClick={handleClose}>
        Cancel
      </Button>
      <Button variant="primary" destructive onClick={handleDelete}>
        Delete Account
      </Button>
    </DialogActions>
  </DialogContent>
</Dialog>
```

### Button Group

```tsx
<ButtonGroup>
  <Button variant="secondary" icon={<AlignLeftIcon />} />
  <Button variant="secondary" icon={<AlignCenterIcon />} />
  <Button variant="secondary" icon={<AlignRightIcon />} />
</ButtonGroup>
```
